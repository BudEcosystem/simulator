# BudSimulator — Optimal, Math-Grounded Remediation Plan

Companion to `bugs.md` (39 findings) and `audit_microbench_data.md` (measurements on this NVIDIA GB10).
**Governing principle:** no hardcoded magic numbers. Every constant traces to (a) a vendor datasheet,
(b) a microbenchmark measured on this box, (c) a published benchmark with citation, or (d) a closed-form
derivation. Where a value must be *fit*, it is fit by a documented, generalizable procedure (the
`differential_evolution` calibration harness) against measured/published data — never hand-typed per case.

Per-unit solutions (RCA → solution → impact → regression → verification) are in §3, produced by the
solution-design workflow and reconciled against the measurements below. §1–§2 are the empirically-derived
backbone; §4 is the cross-coupling/rollout/regression plan.

---

## 1. The empirical foundation (measured, not assumed)

Run on this box (NVIDIA GB10, bf16, torch CUDA). Full data + method: `audit_microbench_data.md`.

| Quantity | Measured | Consequence |
|---|---|---|
| Memory bandwidth (streaming) | 206–211 GB/s = **73–77%** of 273 peak | `eta_mem` ≈ 0.75 (LPDDR5X), **not 1.0** |
| GEMM achievable peak | **85 TFLOPS** = 34% of configured 250 | `eta_compute` must derate to measured achievable |
| Roofline shape | `throughput = min(AI·205, 85)`, ridge AI≈419 | per-op `max(compute,mem)` already encodes shape → **dead TC-eff code is removable, not patchable** |
| Kernel-launch floor | **8.3 µs/op** | grounds `t_launch` (GB10 decode calib 4.56µs ✓) |
| Power | idle 7 W → 96%-util 73 W | power is `idle + util·(TDP−idle)`, **not** fixed 0.41·TDP |

### Keystone end-to-end validations (1.5B Llama, real weights, on GB10) — the regression anchors
- **Decode (memory-bound):** measured 70% MBU, TPOT≈15.9 ms. Sim default `eff=1.0` → **−30%** (too fast).
  Calibrated `eta_mem` → **±7%**. *Confirms C2; proves the fix and its magnitude.*
- **Prefill (compute-bound):** measured ~27% MFU (60–68 TFLOPS). Sim now (`250×0.85`) → **−68 to −72%**
  (≈3× too fast). `eta_compute` fit to prefill → matches. *Confirms C3.*
- **Coupling proof:** at 8k ctx, toggling full-S² vs causal-S²/2 attention shifts the error by ~11 pts —
  the calibrated `eta_compute` *absorbs* the FLOP-counting convention. **⇒ structural fixes first, then
  calibrate against the corrected structure** (see §4).

These two anchors (decode −30%, prefill −3×) are the headline accuracy bugs and are now reproduced on
real silicon with a real model. They are the keystone tests in the master regression suite (§4.3).

## 2. The master fix is one coherent mechanism, not 39 patches

The dominant errors (C1, C2, C3, C4, M4b, L0, L3, M14) are all facets of the same thing: **the roofline
is physically correct in *structure* but runs with peak specs and unity efficiency.** The optimal,
generalizable fix is a layered, calibration-based correction — every layer grounded:

1. **Correct the structural inputs** (so the roofline reflects real work):
   - C1/C6/C7: **dense bf16 per-chip FLOPS** from datasheets (H100 989, H100-PCIe 756, Max-1550 ~839…),
     single source of truth shared by engine + catalog + training.
   - M4b: **causal attention** counts the S(S+1)/2 triangle, not S² (exact, derived — not 0.5 magic).
   - C4/L3/M3: **delete** the ad-hoc `get_tensor_core_efficiency_factor` / `get_shape_dependent_op_intensity`
     / `OPERATION_CHARACTERISTICS` and the flat 0.85 — the roofline already encodes shape via AI (§1).
   - L0: `tf32` compute multiplier 2 (A100: 156 vs 312 TFLOPS, datasheet).
2. **Apply per-device efficiency** `eta_mem`, `eta_compute`, `t_launch`, `per_stream` — **fit** by the
   existing `validation/inference_calibration.py` harness against a measured/published MBU·MFU grid,
   per device. Non-measured devices anchored by **memory technology** (HBM3 ~0.85, HBM2e ~0.8,
   LPDDR5X ~0.75 measured, GDDR ~0.75) and **published cuBLAS/CUTLASS large-GEMM MFU** (~0.7–0.85 of
   *dense* peak), with the citation recorded. No per-device hand-typed constants.
3. **Calibration is fit AFTER step 1** (the coupling proof): otherwise `eta` silently absorbs the
   structural error and won't generalize to other models/contexts.

This is why C5 (18 unsimulatable devices) and the source-of-truth split fall out for free: once the
engine builds a `System` from the same (dense, single-source) hardware row the catalog serves, every
listed device is simulatable and shows the spec it computes with.

### Unit: hardware-single-source-dense-flops-full-simulatability

**Approach.** The original unit conflated three separable problems and got the physics wrong on two of them; this revision splits and re-grounds it. VERIFIED against the actual code and DB: (1) the compute path is compute_time = num_ops·bit_mult/op_per_sec with op_per_sec=flops/2 (system.py:100) and GEMM.get_num_ops = B·M·N·K MACs (operators.py:123-125) — so compute_time = 2·MACs/flops_raw, exactly as the critique states, NOT the inverted form the original wrote; (2) the engine today resolves Flops from STATIC configs only (set_hardware_db_path is never called at runtime — grep-verified), so H100_GPU=1979 (sparse) is what the engine actually simulates, yielding a physically-impossible implied 155% MFU vs the true 989.5 TFLOPS dense bf16 peak on a measured Llama-2-7B prefill; (3) get_tensor_core_efficiency_factor returns a CONSTANT 0.85 in the real GEMM/Logit/Attend path (the shape-dependent branches are dead code — get_dimensions() returns a list for GEMM and a 3-tuple for Logit, both failing the isinstance(tuple)/len>=4 guards — empirically verified across skinny and square shapes), so there is a fixed 1/0.85=1.176x double-correction, not the variable 0.4–0.95 the critique feared. The fix therefore is: (F1) audit the DB to dense bf16 per-chip peaks with a per-device sourced correction table and DEMOTE every real_values=0 placeholder out of the simulatable set rather than enabling it; (F2) make the engine resolve from the DB as the single source by calling set_hardware_db_path at startup, AFTER F1, so no shared name silently regresses; (F3) delete the dead tc_eff/shape-intensity double-correction so the roofline is clean before any eta calibration; (F4) repoint parallelism_optimizer.HARDWARE_CONFIGS at the same DB. Every numeric constant traces to a vendor datasheet; verification uses the engine's own closed-form (2·MACs/flops on a pure large-M GEMM) and asserts a full pre/post spec snapshot over ALL active devices, not a hoped-for MFU.

**Net effect.** Decode (bs=1, memory-bound): UNCHANGED (byte-identical) on every device — the FLOPS/tc_eff fixes touch only the compute term that loses max(compute,memory) in decode. Prefill / compute-bound (the corrected NVIDIA datacenter parts H100/H200/GH200): latency increases ~2.35x (F1 2.0x dense + F3 1.176x dead-code removal), driving implied MFU from a physically-impossible ~155% down to a measured ~66.5% vs the 989.5 dense peak — now plausible vs published cuBLAS/CUTLASS large-GEMM MFU (0.7-0.85) net of attention/norm overhead. RTX4090 prefill unchanged at the engine (stays 330.3 dense; the bug would have been INTRODUCED by F2-without-F1, now prevented). 19 DB-only placeholder devices: change from 'silently simulatable with garbage flops' to 'explicitly gated (raise/estimated)'. Memory footprint and KV-cache math: unchanged (no FLOPS dependence). Power/energy on compute-bound prefill: rises toward realistic (higher compute_utilization), magnitude to be re-snapshotted, not assumed.

**Couplings.** F1 and F2 are strictly ordered and MUST ship together: F2 (wire DB to engine) is only safe AFTER F1 (correct DB to dense + gate placeholders), otherwise wiring the DB regresses RTX4090 (330->661) and activates Cerebras/Groq placeholders in the engine. F1 and F3 COMPOUND on compute-bound latency: F1 doubles H100 prefill (1979->989.5) and F3 multiplies by a further 1/0.85=1.176x; combined ~2.35x — both are correct and both must be measured in the post snapshot (do not attribute the whole move to one). F3 is a HARD PREREQUISITE for the separate eta-calibration unit (per audit_microbench_data.md sec 6 plan-of-record): if eta_compute is fit by differential_evolution while the dead 0.85 is still applied, every fitted eta absorbs the 0.85 and stops being device-portable. F1's placeholder gate COUPLES to the power/cost endpoints: gating real_values=0 devices out of simulatability means power/energy/recommendation for those devices now error (or are marked estimated) instead of returning fabricated numbers — a deliberate, surfaced behavior change. The power model (serving.py estimate_power consuming compute_utilization) is coupled to F1+F3: doubling+1.176x prefill compute_time raises compute_utilization on compute-bound prefill, which raises energy/token; the power endpoint output must be re-snapshotted and is the subject of a separate finding (not silently assumed unchanged). F4 depends on F1/F2 (it delegates to the corrected central source).

### Unit: causal-masking-prefill-attention

**Approach.** Prefill Logit/Attend currently count the FULL S×S (per shard M×N) attention rectangle, but a causal decoder only computes the lower triangle. The correct structural factor is the exact triangle-count ratio (a closed-form derivation, not a fitted constant). The prior proposal had a hard correctness bug: under sequence parallelism the square-prefill op is built with M=input_sequence_length (full logical query length) and N=input_sequence_length//sp (a KEY SHARD, not a causal key budget), so the naive f=1-(M-1)/(2N) goes NEGATIVE for sp>1 (verified: S=4096, sp=4 -> f=-1.0). The revised design (a) derives the factor from the LOGICAL causal geometry per builder, never from raw confounded M,N, and clamps to [0,1] defensively; (b) carries causality as dedicated prefill OpTypes (Logit_PREFILL/Attend_PREFILL) emitted ONLY by causal decoder-LM square/MLA/local prefill builders, so analyse_model gets the causal signal from op_type (it has no model_config.is_decoder), bidirectional/encoder/vision prefill keeps plain Logit/Attend (f=1), and the chunked builder uses its own valid 1-(M-1)/(2N) (M<=N there); (c) scales num_ops AND op_intensity/boundedness consistently in the operator; (d) deletes the dead get_tensor_core_efficiency / get_shape_dependent_op_intensity blocks (proven dead for attention) per the GB10 audit rather than layering the causal factor under unsourced magic constants. Structural FLOP fix lands BEFORE eta_compute is re-fit (audit plan-of-record).

### Unit: Collective / communication cost modeling

**Approach.** The collective model (genz/collective_times.py) is an alpha-beta (latency + data/bandwidth) ring/tree closed form (physically sound) wrapped in an UNSOURCED multiplicative overhead stack (compute_scale_aware_overhead: base_overhead 1.08-1.45, size_factor 1.0-1.15, congestion delta 0.07-0.18, straggler epsilon 0.03-0.08) plus a per-vendor magic table (COMM_EFFICIENCY_TABLES) that is never wired into the live alpha-beta path. #### M1-pp-latency-throughput
- **Root cause:** The critique is correct and the original design misidentified what model_df latency represents. get_language_model.py:471-484 (and the prefill builder 460-491) emits repeat(PP-1)*layers_per_stage + a Message-Pass op + layers_last_stage, i.e. ALL num_decoder_layers are counted exactly once. Therefore decode_latency/prefill_latency from model_df.sum() is the FULL-MODEL single-forward latency, NOT one pipeline stage. The current code at llm_decode.py:177-179 / llm_prefill.py:108-110 computes stage_latency=full/PP and total_latency=full+(PP-1)*stage = full*(2-1/PP), which is ~2x the true full-model latency, and thrpt=1000*B/total uses that inflated denominator. Also ceil(L/PP) makes stages unbalanced when L%PP!=0, so latency/PP is not the true max-stage time, and the graph's Message Pass op (inter-stage activation transfer) is already inside model_df, so dividing by PP wrongly amortizes the comm op into a stage.
- **Solution:** Reframe using the GPipe steady-state with the CORRECT identification that model_df = full-model forward across all stages, and that the per-stage compute is full_compute/PP only in the balanced case. Define per-stage latency from the ACTUAL graph, not full/PP: tag each stage's layer-sum during graph construction is invasive; the minimal correct closed form uses balanced-stage approximation with the true max-stage layer count. (1) Single-request LATENCY (TTFT/TPOT for one sequence traversing the pipeline) = full-model forward latency = decode_latency (the graph already serializes all stages once, including the Message Pass). So total Latency for PP>1 = decode_latency (NOT *(2-1/PP)). The current *(2-1/PP) is wrong and must be removed. (2) THROUGHPUT under microbatching (GPipe steady state) = num_microbatches tokens drained at the bottleneck-stage rate: thrpt = 1000 * batch_size / (full_model_latency * (1 + (PP-1)/num_microbatches)) where the bubble fraction is (PP-1)/(num_microbatches+PP-1). With the unbalanced split, the bottleneck stage time = full_model_latency * (max_layers_per_stage / total_layers); compute max_layers_per_stage = ceil(L/PP). So stage_latency = full_model_latency * ceil(L/PP)/L. Throughput = 1000 * batch_size / (PP * stage_latency + (num_microbatches-1) * stage_latency) for the GPipe fill+drain. This removes both the 2x latency inflation and the balanced-split assumption, and does NOT double-count Message Pass (it stays in full_model_latency which equals the sum of all stages including comm).

(2) max_layers_per_stage = ceil(num_decoder_layers/PP): SOURCE: the graph's own split at get_language_model.py:472, derived (not invented). (3) No fitted constants. There is NO measured PP>1 anchor in the grid (critique correct: INFERENCE_BENCHMARKS has only PP=1), so this fix is validated by INTERNAL CONSISTENCY (PP=1 reduces to existing result) and by the closed-form GPipe derivation, with a comment marking that a measured PP>1 anchor is needed before claiming sub-10% PP accuracy.
